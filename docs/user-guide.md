# DevTracker 사용 가이드

> 야나두 개발팀을 위한 업무/배포 관리 시스템
> URL: https://devtracker-dusky.vercel.app

---

## 1. 로그인

### 접속 및 로그인

1. 브라우저에서 DevTracker URL 접속
2. 이메일과 비밀번호 입력 후 **로그인** 클릭
3. 대시보드로 자동 이동

### 회원가입

1. 로그인 화면 하단의 **"계정이 없으신가요? 회원가입"** 클릭
2. 이름, 이메일, 비밀번호(6자 이상) 입력
3. **회원가입** 클릭

### 테스트 계정

| 역할 | 이름 | 이메일 | 비밀번호 |
|------|------|--------|----------|
| 관리자 | Ted | withwooyong@yanadoocorp.com | yanadoo123 |
| 멤버 | Harrison | kookyh@yanadoocorp.com | yanadoo123 |

> 전체 11개 계정 모두 비밀번호: `yanadoo123`

## 2. 대시보드

로그인 후 가장 먼저 보이는 화면입니다.

### 화면 구성

- 모든 프로젝트가 카드 형태로 표시됩니다
- 각 카드에 **프로젝트 키**, **이름**, **이슈 수**, **배포 수**가 표시됩니다
- 카드를 클릭하면 해당 프로젝트의 이슈 목록으로 이동합니다

### 사용 예시

```
[DEV] 야나두 개발
"야나두 메인 서비스 개발 프로젝트"
이슈 8개 | 배포 1개
```
→ 이 카드를 클릭하면 DEV 프로젝트의 이슈 목록으로 이동

## 3. 프로젝트 관리

### 프로젝트 목록 보기

1. 왼쪽 사이드바에서 **프로젝트** 클릭
2. 모든 프로젝트가 목록으로 표시됩니다

### 새 프로젝트 만들기

1. 프로젝트 페이지에서 **새 프로젝트** 클릭
2. 양식 작성:
   - **프로젝트 이름**: 예) "야나두 앱 개발"
   - **프로젝트 키**: 영문 대문자 2~10자, 예) "APP" (이슈 번호에 사용됨: APP-1, APP-2...)
   - **설명**: 선택 사항
3. **생성** 클릭

### 사용 예시

```
프로젝트 이름: 야나두 앱 개발
프로젝트 키:   APP
설명:         모바일 앱 개발 프로젝트
```
→ 생성 후 이슈 번호가 APP-1, APP-2... 형태로 자동 부여됩니다

### 프로젝트 설정 편집

프로젝트 상단 탭에서 **설정**을 클릭하면 프로젝트 메타 정보를 편집할 수 있습니다.

- **설명** — 언제든 수정 가능 (최대 2000자)
- **GitHub 레포지토리** — `owner/repo` 형식으로 입력. 비워두면 해당 프로젝트는 webhook 연동 대상에서 제외됩니다. (예: `withwooyong/devtracker`)
- **Webhook Secret** — 프로젝트 전용 HMAC secret(16~256자). 설정되면 이 프로젝트의 webhook은 전역 `GITHUB_WEBHOOK_SECRET` 대신 이 값으로 서명을 재검증합니다. 값은 저장 후 다시 표시되지 않으며(서버에서만 보관), 배지로 **설정됨/미설정** 상태만 확인할 수 있습니다. "제거"로 언제든 비울 수 있습니다.

**편집 권한**: ADMIN 역할 또는 해당 프로젝트를 만든 사용자만 저장할 수 있습니다. 권한이 없으면 읽기 전용 안내가 표시됩니다.

> 프로젝트 **키**는 생성 후 변경할 수 없습니다. 기존 이슈 번호 체계를 보호하기 위함입니다.

## 4. 이슈 관리

### 4-1. 이슈 목록

프로젝트 클릭 시 이슈 목록이 표시됩니다. 상단에 탭이 있습니다:
- **이슈 목록** — 테이블 형태
- **칸반 보드** — 드래그 앤 드롭 보드
- **스프린트** — 스프린트 목록 및 번다운
- **배포 이력** — 배포 기록
- **설정** — 프로젝트 메타 편집 (권한 있는 사용자만)

#### 필터 사용하기

이슈 목록 상단의 필터 바에서 조건을 설정할 수 있습니다:

| 필터 | 옵션 | 설명 |
|------|------|------|
| 검색 | 텍스트 입력 | 이슈 제목/설명에서 검색 |
| 상태 | 모든 상태 / 할 일 / 진행 중 / 리뷰 중 / 완료 | 이슈 상태별 필터 |
| 우선순위 | 모든 우선순위 / 긴급 / 높음 / 보통 / 낮음 | 우선순위별 필터 |
| 담당자 | 모든 담당자 / 팀원 이름 | 담당자별 필터 |

#### 필터 저장하기

자주 사용하는 필터 조합을 저장할 수 있습니다:

1. 원하는 필터를 설정합니다 (예: 상태=할 일, 우선순위=긴급)
2. **필터 저장** 버튼이 나타납니다
3. 클릭 후 필터 이름 입력 (예: "긴급 할 일")
4. 팀원과 공유하려면 **공유** 체크
5. **저장** 클릭

#### 저장된 필터 사용하기

1. **저장된 필터** 드롭다운 클릭
2. 원하는 필터 선택 → 즉시 적용
3. 내가 만든 필터는 X 버튼으로 삭제 가능

### 사용 예시

```
시나리오: "내가 담당하고 있는 진행 중인 이슈만 보고 싶다"

1. 상태 필터 → "진행 중" 선택
2. 담당자 필터 → 내 이름 선택
3. "필터 저장" 클릭 → 이름: "내 진행 중 이슈" → 저장

→ 다음부터 "저장된 필터" 드롭다운에서 한 번에 적용 가능
```

### 4-2. 이슈 생성

1. 이슈 목록에서 **이슈 생성** 클릭
2. 양식 작성:

| 필드 | 필수 | 설명 |
|------|------|------|
| 제목 | O | 이슈 제목 |
| 설명 | X | 리치 텍스트 에디터 (서식 지원) |
| 상태 | O | 할 일(기본값) / 진행 중 / 리뷰 중 / 완료 |
| 우선순위 | O | 보통(기본값) / 낮음 / 높음 / 긴급 |
| 담당자 | X | 팀원 선택 또는 미지정 |
| 마감일 | X | 날짜 선택 |
| 라벨 | X | 버그, 기능, 개선, 긴급, 문서 중 선택 (복수 가능) |

3. **이슈 생성** 클릭

#### 리치 텍스트 에디터 사용법

설명과 댓글 작성 시 서식을 적용할 수 있습니다:

| 버튼 | 기능 | 단축키 |
|------|------|--------|
| **B** | 굵게 | Ctrl+B |
| *I* | 기울임 | Ctrl+I |
| ~~S~~ | 취소선 | Ctrl+Shift+X |
| H1 / H2 / H3 | 제목 | - |
| • 목록 | 글머리 기호 목록 | - |
| 1. 목록 | 번호 목록 | - |
| <> | 코드 블록 | - |
| ❝ | 인용문 | - |
| ─ | 구분선 | - |

### 사용 예시

```
제목:     결제 API 오류 수정
설명:     (리치 에디터로 작성)
          ## 재현 방법
          1. BC카드로 결제 시도
          2. 3D Secure 인증 후 에러 발생

          ## 기대 결과
          결제가 정상 완료되어야 함
상태:     할 일
우선순위: 긴급
담당자:   Dior
라벨:     버그, 긴급
```

### 4-3. 이슈 상세

이슈 제목을 클릭하면 상세 페이지로 이동합니다.

#### 왼쪽 영역 — 내용

- **제목과 설명**: "편집" 버튼으로 수정 가능
- **탭 전환**:
  - **댓글** — 댓글 목록 + 댓글 작성 폼
  - **활동** — 이슈 변경 이력 (상태 변경, 담당자 변경 등)
  - **전체** — 댓글과 활동을 시간순으로 합쳐서 표시

#### 오른쪽 사이드바 — 속성 변경

드롭다운을 변경하면 **즉시 저장**됩니다:
- **상태** 변경 → 할 일 / 진행 중 / 리뷰 중 / 완료
- **우선순위** 변경 → 낮음 / 보통 / 높음 / 긴급
- **담당자** 변경 → 팀원 선택

#### 댓글 작성

1. "댓글" 탭 선택
2. 텍스트 입력 (리치 에디터)
3. **댓글 작성** 클릭

#### 활동 로그 보기

1. "활동" 탭 클릭
2. 모든 변경 이력이 타임라인으로 표시:
   - "Ted님이 상태를 변경했습니다: 할 일 → 진행 중"
   - "Harrison님이 댓글을 작성했습니다"
   - "Ted님이 담당자를 변경했습니다"

### 사용 예시

```
시나리오: 이슈 작업을 시작하면서 상태 업데이트

1. 이슈 상세 페이지 열기
2. 오른쪽 사이드바에서 상태를 "진행 중"으로 변경
3. 댓글 탭에서 "작업 시작합니다. 예상 완료: 내일 오후" 입력
4. "댓글 작성" 클릭

→ 활동 탭에 "상태를 변경했습니다: 할 일 → 진행 중" 자동 기록
→ 전체 탭에서 댓글과 상태 변경을 시간순으로 확인 가능
```

## 5. 칸반 보드

### 기본 사용법

1. 프로젝트에서 **칸반 보드** 탭 클릭
2. 4개 컬럼으로 이슈가 분류됩니다:
   - **할 일** (회색) — TODO
   - **진행 중** (파란색) — IN_PROGRESS
   - **리뷰 중** (노란색) — IN_REVIEW
   - **완료** (초록색) — DONE

### 드래그 앤 드롭

- 이슈 카드를 **클릭한 채로 드래그**하여 다른 컬럼으로 이동
- 같은 컬럼 내에서 순서 변경도 가능
- 이동하면 상태가 자동으로 변경됩니다

### 카드 정보

각 카드에 표시되는 정보:
- 이슈 제목 (클릭하면 상세 페이지)
- 이슈 번호 (예: DEV-3)
- 우선순위 배지
- 담당자 이름 + 아바타

### 사용 예시

```
시나리오: 코드 리뷰가 완료되어 이슈를 "완료"로 이동

1. 칸반 보드 탭 클릭
2. "리뷰 중" 컬럼에서 해당 이슈 카드를 찾기
3. 카드를 클릭한 채로 "완료" 컬럼으로 드래그
4. 카드를 놓으면 자동으로 상태가 "완료"로 변경

→ 이슈 상세의 활동 로그에 "상태를 변경했습니다: 리뷰 중 → 완료" 기록
```

## 6. 배포 관리

### 배포 이력 보기

1. 프로젝트에서 **배포 이력** 탭 클릭
2. 환경별 필터: **전체** / **DEV** / **STAGING** / **PROD**
3. 각 배포 기록에 버전, 환경, 상태, 배포자, 변경사항이 표시됩니다

### 배포 기록하기

1. **배포 기록** 클릭
2. 양식 작성:

| 필드 | 필수 | 설명 | 예시 |
|------|------|------|------|
| 버전 | O | 배포 버전 | v1.2.3 |
| 환경 | O | DEV / STAGING / PROD | PROD |
| 설명 | X | 배포 요약 | 결제 버그 수정 핫픽스 |
| 변경사항 | X | 상세 변경 내역 (마크다운) | - 결제 API 오류 수정 |

3. **배포 기록** 클릭

> **참고**: PROD(운영) 배포 기록은 관리자(ADMIN)만 생성할 수 있습니다.

### 배포 상태

| 상태 | 설명 |
|------|------|
| 대기 | 배포 예정 |
| 진행 중 | 배포 진행 중 |
| 성공 | 배포 완료 |
| 실패 | 배포 실패 |
| 롤백 | 이전 버전으로 복구 |

### 사용 예시

```
버전:     v1.3.0
환경:     PROD
설명:     2024년 4월 정기 배포
변경사항:
  - 기능 추가: 회원 프로필 이미지 업로드
  - 버그 수정: BC카드 결제 오류
  - 개선: 대시보드 로딩 속도 최적화
```

## 7. 스프린트 (Phase 2-1)

기간 기반 작업 단위로 이슈를 묶어 진행 상황을 추적합니다.

### 7-1. 스프린트 생성

1. 프로젝트 페이지 상단 탭 → **스프린트** 클릭
2. 우측 상단 **스프린트 생성** 클릭
3. 이름, 목표, 시작일/종료일 입력 → **생성**

### 7-2. 이슈 할당 + 진행

1. 스프린트 상세 페이지에서 **이슈 추가** → 백로그(미할당) 이슈 목록에서 선택
2. **스프린트 시작** 버튼 → 상태가 PLANNED → ACTIVE로 전환
3. 이슈 상태를 DONE으로 바꾸면 번다운 차트의 실제선이 반영됨
4. **스프린트 완료** 버튼 → 상태가 COMPLETED로 전환

### 7-3. 번다운 차트

- 이상치(회색 점선): 기간 동안 균등하게 이슈가 감소하는 이상적 진행
- 실제(파란 실선): `completedAt` 기준의 실제 완료 흐름
- 오늘(주황 점선): 현재 시점 마커

## 8. 알림 (Phase 2-2)

### 8-1. 알림 받기

화면 오른쪽 상단의 **종 아이콘 🔔**에 미읽음 알림 개수가 빨간 배지로 표시됩니다. 다음 상황에 자동 전송:

- 이슈가 나에게 할당됨
- 내가 담당인 이슈의 상태가 바뀜
- 내가 담당 또는 보고자인 이슈에 댓글이 달림
- 내가 담당 이슈가 포함된 스프린트가 시작/완료됨

### 8-2. 알림 읽기

1. 종 아이콘 클릭 → 드롭다운 열림 (최신 30건)
2. 알림 클릭 → 해당 이슈/스프린트로 이동 + 자동으로 읽음 처리
3. 우측 상단 **모두 읽음** 버튼으로 일괄 처리

## 9. 파일 첨부 (Phase 3-1)

이슈에 이미지/PDF/Office 문서/Zip 등을 첨부할 수 있습니다.

### 9-1. 업로드

1. 이슈 상세 페이지의 **첨부파일** 섹션으로 스크롤
2. 드래그앤드롭 또는 **클릭하여 선택** → 파일 선택
3. 업로드 진행 중 표시 → 완료 시 목록에 추가

**제약**:
- 최대 4MB (Vercel 서버리스 제한)
- 이슈당 최대 20개
- 허용 형식: 이미지, PDF, Zip, Text, Office 문서

### 9-2. 다운로드 / 미리보기

- 이미지는 썸네일로 즉시 미리보기 (인증 확인 후 서버 프록시 경유)
- 파일명 클릭 → 새 탭에 인라인 표시
- 삭제는 **본인이 업로드한 파일** 또는 **ADMIN**만 가능

## 10. GitHub 연동 (Phase 3-2)

### 10-1. PR에 이슈 키 포함

PR 제목이나 브랜치명에 이슈 키(`DEV-123` 형태)를 포함하면 자동으로 이슈에 연결됩니다.

예시:
- PR 제목: `[DEV-42] 로그인 버그 수정`
- 브랜치명: `feature/dev-42-login-fix`

### 10-2. 자동 동작

| GitHub 이벤트 | DevTracker 동작 |
|---------------|-----------------|
| PR Opened | 이슈 상세의 **GitHub** 섹션에 PR 링크 + "열림" 상태 표시 |
| PR Closed | 상태 "닫힘"으로 변경 |
| PR Merged | 이슈 상태를 **DONE**으로 자동 전환 + Activity 기록 |
| Push (commit) | 커밋 메시지에 이슈 키가 있으면 **GitHub** 섹션에 커밋 링크 추가 (이슈 상태는 변경하지 않음) |

### 10-3. 주의사항

- Webhook은 관리자가 설정한 단일 GitHub 레포에만 연결됨 (현재: `withwooyong/devtracker`)
- PR에 이슈 키가 없으면 자동 연결은 되지 않음 (PR 제목 기준)
- Push 이벤트는 커밋 메시지 기준으로 연결. 브랜치 삭제나 빈 push는 조용히 무시됩니다
- 이슈 상태 변경(DONE 전환)은 PR Merge에서만 발생 — push 이벤트는 커밋 링크만 추가합니다

### 10-4. GitHub 계정 연결(사용자 매핑)

사이드바 하단의 **내 프로필**에서 본인의 **GitHub 로그인**을 등록하면, 본인이 올린 PR이 머지될 때 이슈 상태 변경(STATUS_CHANGED) Activity가 본인 명의로 기록됩니다.

- 미등록 상태에서는 이슈 작성자(reporter) 명의로 폴백 기록됩니다.
- 한 번 매칭되면 GitHub의 숫자 ID가 자동 저장되어, 이후에 GitHub 로그인을 변경해도 계속 매핑이 유지됩니다.
- 다른 사용자가 이미 사용 중인 GitHub 로그인은 등록할 수 없습니다(409 에러 표시).

## 자주 묻는 질문

**Q: 비밀번호를 잊어버렸어요**
A: 관리자(Ted)에게 문의하세요. 현재 비밀번호 재설정 기능은 없습니다.

**Q: 이슈 번호는 어떻게 정해지나요?**
A: 프로젝트 키 + 자동 증가 번호입니다. (예: DEV-1, DEV-2, DEV-3...)

**Q: 삭제한 이슈를 복구할 수 있나요?**
A: 현재 복구 기능은 없습니다. 삭제 전 신중하게 판단해주세요.

**Q: 필터를 공유하면 누가 볼 수 있나요?**
A: 같은 프로젝트의 모든 팀원이 사용할 수 있습니다. 삭제는 만든 사람 또는 관리자만 가능합니다.

**Q: 칸반 보드에서 이슈가 안 보여요**
A: 칸반 보드는 최대 100개 이슈까지 표시됩니다. 이슈가 많으면 이슈 목록 탭에서 필터를 사용하세요.
