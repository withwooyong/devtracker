# /ted-run 사용 가이드

구현 → 코드리뷰 → 빌드검증 → 커밋/푸시를 자동으로 수행하는 파이프라인 명령어.

## 기본 사용법

```
/ted-run                          # 현재 작업계획서 기반 전체 파이프라인
/ted-run <작업계획서.md>           # 특정 작업계획서 지정
/ted-run --from <step>            # 특정 단계부터 재개
```

### `--from` 재개 옵션

| 옵션 | 시작 단계 | 전제 조건 |
|------|----------|----------|
| `--from review` | Step 2 코드리뷰 | 구현 완료 상태 |
| `--from build` | Step 3 빌드검증 | 리뷰 완료 상태 |
| `--from commit` | Step 4 커밋/푸시 | 빌드 완료 상태 |

## 파이프라인 4단계

### Step 1: 구현 (Implement)

작업계획서(`.md`)를 읽고 코드를 구현한다.

- 독립적인 Phase/모듈이 있으면 **전문가 팀을 병렬 실행** (Agent)
- 단일 파일 수정이면 직접 구현
- 언어별 전문 에이전트 자동 선택 (TypeScript, Kotlin, Java, Python, Go, Rust)

### Step 2: 코드 리뷰 (Review)

구현 완료 후 자동으로 코드 리뷰를 실행한다.

| 심각도 | 조치 |
|--------|------|
| CRITICAL / HIGH | 즉시 수정 후 재리뷰 |
| MEDIUM | 수정 후 진행 (재리뷰 불필요) |
| LOW | 기록만 하고 진행 |

CRITICAL/HIGH가 0건이 되어야 다음 단계로 진행한다.

### Step 3: 빌드 + 실행 검증 (Build & Run)

1. **컴파일 빌드** — 언어별 빌드 명령 실행 (`pnpm build`, `./gradlew compileKotlin` 등)
2. **실행 검증** — 로컬에서 서버 기동하여 런타임 오류 확인

빌드 오류 발생 시 언어별 전문 에이전트가 자동 수정을 시도한다.

### Step 4: 커밋 / 푸시 (Ship)

1. HANDOFF.md 업데이트
2. 변경 파일 스테이징 (민감 파일 자동 제외)
3. 한글 커밋 메시지 작성 + `Co-Authored-By` 포함
4. 사용자 확인 후 푸시

## 진행 현황 표시

각 단계 완료 시 아래 형태로 보고된다:

```
| Step | 상태 | 비고 |
|------|------|------|
| 1. 구현 | ✅ 완료 | 3개 팀 병렬, 파일 N개 |
| 2. 코드 리뷰 | ✅ 완료 | CRITICAL 0, HIGH 0 |
| 3. 빌드 검증 | ✅ 완료 | tsc --noEmit 성공 |
| 4. 커밋/푸시 | ⬜ 대기 | 사용자 확인 필요 |
```

## 사용 예시

### 전체 파이프라인 실행

```
/ted-run PLAN.md
```

작업계획서를 기반으로 구현 → 리뷰 → 빌드 → 커밋을 한 번에 수행.

### 코드리뷰부터 재개

```
/ted-run --from review
```

이미 구현은 끝났고 리뷰/빌드/커밋만 진행할 때.

### 커밋만 실행

```
/ted-run --from commit
```

리뷰와 빌드까지 통과한 상태에서 커밋/푸시만 진행할 때.

## 주의사항

- 코드 리뷰에서 "기존 코드 이슈"와 "이번 변경 이슈"를 구분한다
- 빌드 검증은 반드시 코드 리뷰 후에 실행한다 (리뷰 수정이 빌드를 깨뜨릴 수 있음)
- 커밋 시 민감 파일(`.env`, DB 정보 등)이 포함되지 않도록 자동 확인한다
- Step 4 진입 시 "커밋/푸시 진행할까요?" 확인을 받는다
